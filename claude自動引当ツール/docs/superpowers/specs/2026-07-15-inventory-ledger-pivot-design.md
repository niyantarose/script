# 在庫管理ツールへのピボット設計 — 足し引き台帳を核とした在庫正本化

日付: 2026-07-15
対象: claude自動引当ツール (Flask / SQLite / VPS systemdタイマー)

## 背景と目的

受注引当ツールとして始まったが、在庫管理ツールへピボットする。目指す姿:

- Yahoo在庫をAPIで自動取得(既存機能)し、注文で自動的に在庫を引き、キャンセルで自動的に戻す
- ツール内の在庫台帳を段階的に「正本」とし、モール側へ在庫数を書き戻す
- 複数モール対応(優先順: Yahoo → Amazon → Qoo10 → メルカリShops/TikTokは任意)
- 棚卸・ロケーション管理(日本の保管場所のみ・棚番号レベル)

**設計の最優先事項はユーザーの明示的な指示により「引当機能の高度化」ではなく「在庫の足し引きが正確にできること」。**

**本システムは日本国内在庫のみを管理対象とする。** 大邱など海外側の在庫・EMS輸送中在庫は台帳の対象外(既存のEMS管理機能のまま)であり、日本到着時点で初めて「入庫」として台帳に記録される。

## 役割分担(確定事項)

| 領域 | 担当 |
|---|---|
| 受注処理(出荷・送り状・メール) | GoQシステム(現行のまま) |
| 在庫の正本・足し引き・棚卸・モール書き戻し | このツール |

## 前提となる既存資産

- Yahoo API連携稼働中: 注文取込(`import_yahoo_orders`)、在庫差分/全件取込、OAuth(quiet再認証付き)
- VPS systemdタイマー: zaiko-yahoo-orders / zaiko-yahoo-stock-diff / zaiko-yahoo-stock-full / zaiko-run-all-imports
- `Inventory` モデルに `location`(棚番号)・`reserved_qty`・`available_qty` あり
- 引当エンジン(`services/allocation.py`、入荷日ベース)は既存機能として維持(今回の核ではない)
- データ取込4系統: Yahoo API / Google Sheets / Cloudike WebDAV / Excel

## フェーズ構成

| フェーズ | 内容 | 完了時に出来ること |
|---|---|---|
| 1 | 在庫台帳(足し引き)の正本化 + Yahoo注文の自動マイナス/キャンセル戻し | 在庫のすべての増減が台帳に記録され、注文・キャンセルに自動追従 |
| 2 | Yahoo在庫書き戻し(off→dry_run→allowlist→full) | ツールの在庫数がYahooへ自動反映(売り越し防止) |
| 3 | 棚卸・ロケーション管理 | 棚別在庫一覧、棚卸セッション、差異調整 |
| 4 | マルチモール(Amazon→Qoo10→他) | 全モールの注文が同一台帳を足し引き、各モールへ書き戻し |

各フェーズは単体で価値が出る。フェーズごとに スペック→計画→実装 のサイクルを回す(本設計はフェーズ全体の親設計)。

## データモデル(新規4テーブル)

### 1. stock_transactions(在庫トランザクション台帳) — 設計の核

すべての在庫変動を追記型(append-only)で記録する。

| 列 | 内容 |
|---|---|
| id | PK |
| product_code / product_sub_code | 対象商品(既存Inventoryと同じキー体系) |
| tx_type | receive(入庫) / order_out(注文出庫) / cancel_return(キャンセル戻し) / manual_in / manual_out / adjust(棚卸・訂正) |
| qty | 増減数(＋/−) |
| ref_type / ref_id | 発生元(yahoo_order_item / ems_item / stocktake / manual 等) |
| source_key | 発生元の一意キー(例: `yahoo:注文ID:行番号:order_out`)。**UNIQUE制約で二重計上を構造的に防止** |
| reason / memo | 手動操作時の理由 |
| created_at | 記録日時 |

- 現在庫 = 台帳の合計。`Inventory.quantity` は台帳から再計算されるキャッシュ値と再定義する。キャッシュはトランザクション記録時に同期更新し、加えて日次で全件再計算して乖離があればアラート(自己検証)。
- 訂正は行の書き換えではなく逆方向の adjust 行を追加(監査可能性の維持)。

### 2. mall_skus(モール商品コードマッピング)

| 列 | 内容 |
|---|---|
| mall | yahoo / amazon / qoo10 / mercari / tiktok |
| external_code / external_sub_code | モール側商品コード |
| product_code / product_sub_code | 内部商品コード |

- Yahoo分は既存在庫データから1:1で自動シード。
- Phase 4の土台だがPhase 1で先行作成(後からの移行を避ける)。

### 3. stock_push_logs(書き戻し監査ログ)

mall、商品コード、変更前→変更後の数量、モード(dry_run/live)、結果(成功/失敗/スキップ理由)、日時。書き戻しの全操作を記録し、事故時の遡及復旧を可能にする。

### 4. stocktakes / stocktake_lines(棚卸セッション)

- stocktakes: セッション(開始日時、対象=全件 or 棚指定、状態=進行中/確定、確定日時)
- stocktake_lines: 商品、システム在庫(開始時点)、実数、差異、調整済みフラグ
- 確定時に差異分を stock_transactions へ adjust として記録。途中保存可。

## 足し引きの種類と発生元

| 方向 | 種別 | 発生元 | 自動/手動 |
|---|---|---|---|
| ＋ | 仕入れ到着 | EMS明細の日本到着処理(既存フローに接続) | 自動 |
| ＋ | 手動入庫 | 画面から数量入力 | 手動 |
| ＋ | キャンセル戻し | Yahoo注文のキャンセル検知 | 自動 |
| − | 注文出庫 | Yahoo注文の取込時 | 自動 |
| − | 手動出庫 | 破損・サンプル・その他(理由必須) | 手動 |
| ± | 調整 | 棚卸確定・手動訂正 | 手動 |

## 動作フロー

### 注文→自動マイナス、キャンセル→自動プラス(Phase 1)

1. 既存タイマーの注文取込(`_upsert_yahoo_order`)の後段で、未処理の注文行に対し `order_out` を台帳へ記録。
2. 取込時にYahoo側ステータスが「キャンセル」へ変わった注文を検知したら `cancel_return` を記録して在庫を戻す。
3. **出荷済み注文のキャンセルは自動で戻さない。** アラートを出し、返品現物の到着後に手動入庫する。
4. 冪等性: source_key のUNIQUE制約により、同じ注文を何度取り込んでも足し引きは1回だけ。

### 手動の足し引きUI(Phase 1)

商品検索 → 「＋入庫 / −出庫 / 調整」選択 → 数量・理由入力 → 即時台帳記録。1画面完結、スマホ操作可能なシンプルさを優先。商品ごとの増減履歴画面(いつ・何で・何個)を併設。

### Yahoo在庫書き戻し(Phase 2)

環境変数 `STOCK_PUSH_MODE` で4段階制御:

| モード | 動作 |
|---|---|
| off | 書き戻しなし(初期値) |
| dry_run | 更新内容をログ記録のみ |
| allowlist | 指定商品コードのみ実更新(少数試験) |
| full | 全商品対象 |

安全弁: ①1回の同期で更新対象がしきい値(既定50件、環境変数で変更可)超過で停止+アラート、②Yahoo側在庫と台帳の乖離がしきい値(既定±20個、環境変数で変更可)以上の商品はスキップ+アラート。全書き戻しは stock_push_logs に記録。

### 棚卸・ロケーション(Phase 3)

- 棚マスタ + 棚別在庫一覧(既存 `Inventory.location` を活用)。
- 棚卸モード: セッション開始 →(全件 or 棚指定)実数入力 → 差異一覧 → 確定で adjust 記録。

### マルチモール(Phase 4)

- mall_skus で外部コード→内部コード変換し、全モールの注文が同一台帳を足し引き。
- 追加順: Amazon SP-API(自己発送分のみ、FBAはAmazon管理のため対象外)→ Qoo10 → メルカリShops/TikTok(任意)。
- 各モールについて「注文取込→自動マイナス」「キャンセル→戻し」「在庫書き戻し」を同じパイプラインに載せる。

## エラー処理

- API失敗: リトライ後、既存アラート機構(`models/alert.py`)で通知。
- 書き戻し安全弁の発動: アラート通知し、当該同期は停止(手動確認後に再開)。
- 出荷済みキャンセル: 自動処理せずアラート(手動判断)。

## テスト方針

- 台帳ロジック(足し引き・source_key冪等性・キャンセル戻し・調整)にユニットテストを厚く書く。
- 書き戻しは dry_run ログとYahoo実値の突合を試験手順とし、allowlist で少数商品の実試験を経て full へ。
- 棚卸は差異確定→台帳反映→現在庫再計算の一致を検証。

## スコープ外(明示)

- GoQシステムの置き換え(出荷・送り状・メール)
- 海外拠点(大邱等)・EMS輸送中在庫の台帳管理(日本到着時点で入庫として記録。海外側は既存のEMS管理機能のまま)
- FBA在庫の管理
- 引当エンジンの高度化(既存機能の維持のみ)

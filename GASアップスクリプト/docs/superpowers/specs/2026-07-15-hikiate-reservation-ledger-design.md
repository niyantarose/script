# 引当取り置き台帳 設計

日付: 2026-07-15
対象: `Project_24`（引当ファイル）

## 背景

現行の引当は、受注明細の入荷日、EMS番号、引当履歴、消込台帳の出荷日を組み合わせて「どの在庫を使ったか」を推測している。そのため、開始前から棚にあった注文別取り置きと今回到着したEMS在庫が同じ商品コードの場合、過去の注文や出荷が今回便を消費したように見える。

実例では、`KRBLCM16-02` の現物が棚に残っているのに今回便から出荷済み扱いで消え、`OFW300`、`OFW301`、`MZBGD03`、`JNXGD01` は開始前取り置きがあるのに今回便へ再引当された。`YMNGD09` は自動計算分と手入力分が重複した。

受注明細に追加した `取り置き数` と消込台帳メモによる補正は、入荷日ありの在庫消費、出荷可否、全件検算の全経路に接続されておらず、恒久的な正データにはできない。

## 目的

- 注文番号ごとの物理的な取り置き数量を、CSV更新で消えない独立台帳に保持する。
- 開始前取り置き、今回EMS引当、キャンセル戻しの出どころを追跡できるようにする。
- 入荷日だけを根拠に、過去の注文や出荷へ今回便を推測消費させない。
- 初回棚卸後は、通常の受注明細更新と引当実行で自動維持する。
- Yahoo即納在庫と注文番号付き取り置きを混ぜない。
- 同じ処理を再実行しても数量が増えない冪等な処理にする。

## 対象外

- Yahoo即納在庫の初回全数棚卸。
- Yahoo末尾 `a` 在庫を韓国取り寄せ注文の引当供給として使用すること。
- 台湾・中国ルートの在庫管理変更。
- 商品コードの類似一致や、不正な連結コードの自動分割。
- 現物確認なしでキャンセル品をYahoo在庫へ自動加算すること。

## 正とするデータ

### 注文別取り置き

新設する `取り置き台帳` のうち、状態が `取り置き中` の行だけを注文別確保の正とする。受注明細の入荷日、EMS番号、既存の `取り置き数`、消込台帳メモは、移行後は注文別確保数量の正として使用しない。

注文行の今回必要数は次で計算する。

`今回必要数 = max(0, 注文数量 - 取り置き中数量)`

取り置き中数量は、同じ受注番号、商品コード、SKUに紐づく有効行の合計である。

### Yahoo即納在庫

Yahoo即納在庫は、日本ですぐ販売できる自由在庫として扱い、`取り置き台帳` の注文別確保数量へ加えない。Yahoo CSVは比較・確認には使用できるが、EMS引当供給にはしない。

### EMS供給

追跡可能な実EMS番号を持ち、状態が到着済み、商品コードが有効、数量が正数の行だけを供給とする。棚卸行、EMS番号空欄行、未着行は供給へ加えない。

EMS在庫を消費できるのは、そのEMS番号を出どころとして `取り置き台帳` に登録された数量だけである。入荷日一致や出荷日以前という理由だけでは消費しない。

### 商品コード

数値枝番は削除しない。次は相互に独立した商品である。

- `OFW304-1`
- `OFW304-2`
- `OFW305-1`
- `OFW305-2`

販売用SKUの末尾 `a` / `b` を基底コード照合で落とす既存ルールは維持する。注文番号指定の商品、`POEM65`、`RECIPE42` などの特殊商品は通常FIFOと分け、既存の専用判定を維持する。

## 取り置き台帳

`取り置き台帳` は最低限、次の列を持つ。

| 列 | 内容 |
| --- | --- |
| 取置ID | 変更されない一意ID |
| 状態 | 取り置き中 / 発送済み / キャンセル戻し / 手動解除 |
| 受注番号 | 注文の識別子 |
| 商品コード | 表示用の商品コード |
| SKU | 受注行との安定照合用 |
| 取り置き数量 | その行で確保している数量 |
| 取置元種別 | 開始前在庫 / EMS / キャンセル再引当 |
| 元EMS番号 | EMS由来の場合の実EMS番号 |
| 元EMS商品コード | 箱側の元コード。注文番号在庫では受注明細の商品コードと異なるため保持する |
| 元取置ID | キャンセル再引当時の移動元 |
| 登録日時 | 初回登録日時 |
| 更新日時 | 最終状態変更日時 |
| 戻し処理結果 | 未確認 / 現物あり / 再引当済み / Yahoo反映済み / 在庫なし |
| 終了理由・メモ | 手動解除理由など |

数量は正の整数だけを許可する。注文数量を超える取り置き中数量はエラーとし、自動補正しない。

受注行の安定キーは `受注番号 | 正規化商品コード | SKU` とする。行番号はCSV更新で変わるためキーに使用しない。正規化では販売用末尾 `a` / `b` だけを落とし、数値枝番は保持する。

### 状態の意味

- `取り置き中`: 現物が棚にあり、その注文のために確保されている。注文の必要数から差し引く。
- `発送済み`: 取り置き品を発送済み。棚にはないが、出どころと数量の監査記録として残す。
- `キャンセル戻し`: 注文キャンセルで確保先を失った。現物確認と再引当またはYahoo戻しを待つ。
- `手動解除`: 注文は残るが、登録間違い、現物不足などの理由で取り置きを外した。理由入力を必須とする。

### EMS在庫移動台帳

注文別確保とは別に、⑤でYahooへ移した数量を `EMS在庫移動台帳` に記録する。最低限、処理ID、EMS番号、商品コード、数量、移動先、処理日時を持つ。

同じ処理IDは一度だけ有効にし、⑤の再実行でYahoo移動数量を重複させない。取り置き数量の正は `取り置き台帳`、EMSからYahooへの移動証跡の正は `EMS在庫移動台帳` とし、役割を混ぜない。

## 初回移行

1. `部分在庫` と `希望日待ち` の現存注文を `取り置き初期登録` シートへ自動抽出する。
2. 受注番号、商品コード、SKU、注文数量を自動表示する。
3. 利用者が現物を確認し、実際の取り置き数量だけを入力する。
4. 数量ゼロまたは空欄は登録しない。
5. 注文数量超過、重複キー、不正な商品コードがあれば確定を中止する。
6. `初期登録を確定` で、取置元種別 `開始前在庫`、状態 `取り置き中` として台帳へ一括登録する。
7. 同じ初期登録を再実行しても、受注番号・商品コード・SKUの初期行を洗い替えし、重複追加しない。

Yahoo即納在庫は初期登録へ含めない。

## 通常データフロー

### ①受注明細更新

1. GoQの全ステータスCSVを受注明細へ取り込む。
2. 受注番号、商品コード、SKUで `取り置き台帳` と照合する。
3. 注文が継続中なら、取り置き中行をそのまま保持する。
4. 明示的な処理済み・発送済み、または運用上発送済みを意味する消滅を検出したら、対応する取り置き中行を `発送済み` にする。
5. キャンセルを検出したら、対応する取り置き中行を `キャンセル戻し`、戻し処理結果を `未確認` にする。
6. CSV取込や行順変更では台帳を削除しない。

同じ受注番号に複数商品、同一商品複数個がある場合も、商品コード・SKU・数量単位で処理する。

消滅を発送済みとして扱うのは、最新CSVが全ステータスの完全なスナップショットであることを確認できた場合だけにする。ファイル欠損、列不足、取得件数の異常がある場合は自動遷移せず `要確認` に出す。

### キャンセル戻し

1. `キャンセル戻し確認` シートに、受注番号、商品コード、数量、元EMS番号を表示する。
2. 利用者が現物を確認し、`現物あり` または `在庫なし` を選ぶ。
3. `現物あり` の数量は、次回④で同一商品を待つ最古の注文へ、今回EMSより先に再引当する。
4. 再引当した場合は、新しい注文へ取置元種別 `キャンセル再引当` の取り置き中行を作り、元取置IDを残す。元行の戻し処理結果を `再引当済み` にする。
5. 待ち注文がない場合は `Yahoo戻し候補` に出す。Yahooへ実際に加算した後だけ戻し処理結果を `Yahoo反映済み` にする。
6. 現物がなければ戻し処理結果を `在庫なし` にし、Yahooにも他注文にも加えない。

Yahoo在庫は現物確認前に自動加算しない。

### ④引当実行

④はすべてメモリ上で計算と検証を終えてから、一括反映する。

1. 取り置き中数量を各注文へ充当する。
2. 現物確認済みのキャンセル戻しを、同一商品を待つ最古の注文へ再引当する。
3. 注文番号指定の商品を指定注文へ割り当てる。
4. 各注文の今回必要数を再計算する。
5. 今回EMSを不足分へ、既存のP列確定を優先し、その後は古い注文順のFIFOで割り当てる。
6. 新規割当を取置元種別 `EMS`、元EMS番号付きの取り置き中行として台帳へ登録する。
7. どの注文にも割り当てられなかった今回EMS数量だけを日本在庫候補へ出す。

④再実行時は、受注番号、商品コード、SKU、取置元種別、元EMS番号ごとの目標数量を既存台帳へ洗い替えする。追加だけを行わず、同じ割当の二重登録を防ぐ。

### ⑤Yahoo在庫反映

⑤の対象は、④で注文へ割り当てられなかった実EMS余りと、現物確認済みで待ち注文のないキャンセル戻しだけである。Yahoo反映に成功した数量だけを一意な処理ID付きで `EMS在庫移動台帳` へ登録する。失敗またはキャンセル時は台帳へ登録せず、再実行できる状態を保つ。

## 計算順序

同じ商品に複数の供給候補がある場合、次の順に使用する。

1. その注文に既に紐づく取り置き中
2. 現物確認済みキャンセル戻し
3. 注文番号指定在庫
4. 今回到着EMS
5. 余りを日本在庫候補

Yahoo即納在庫はこの順序へ入れない。

## 不変条件と停止条件

次の条件を④確定前に検証する。

1. 取り置き数量は正の整数である。
2. 注文別取り置き中合計は注文数量以下である。
3. 同一取置IDは一行だけである。
4. EMS番号別・商品別の供給数量は、EMS由来の取り置き中、発送済み、戻し未処理、在庫なし確定、Yahoo移動済み、余りの合計と一致する。再引当済みの元行は除外し、移動先の取り置き中行だけを数える。`現物あり` は戻し未処理へ含め、`在庫なし` は再利用できない確定差引きとして別に数える。
5. EMS供給を超える割当を作らない。
6. 注文別の今回必要数は負にならない。
7. 数値枝番を持つ別商品を統合しない。
8. 注文番号指定在庫を通常商品FIFOへ流さない。
9. キャンセル戻しは現物確認前に供給へ加えない。

一件でも違反した場合は台帳、受注明細、引当履歴、出力シートを変更せず、受注番号、商品コード、数量、違反理由を一覧表示する。

## 出力と運用画面

- `取り置き台帳`: 監査用の正データ。通常は直接編集しない。
- `取り置き初期登録`: 初回だけ数量を入力する。
- `キャンセル戻し確認`: 現物あり・在庫なしを選択する。
- `Yahoo戻し候補`: 待ち注文がなくYahooへ戻せる現物を表示する。
- `EMS在庫移動台帳`: ⑤でYahooへ移した数量の監査記録。通常は直接編集しない。
- `部分在庫`、`希望日待ち`、`出荷可能`、`引当待ち`: 取り置き台帳と現受注から毎回再生成する。
- `今回入荷EMSの在庫`: EMS番号ごとの割当先と余りを表示する。
- `全件検算`: 取り置き台帳を含む同じ計算結果から生成する。

状態名は利用者向けに `取り置き中`、`発送済み`、`キャンセル戻し`、`手動解除` を使用する。

## エラー処理

- 初回登録の入力数量が注文数量を超える場合は確定しない。
- 取り置き中の注文がCSVに見つからず、発送ともキャンセルとも断定できない場合は自動終了せず `要確認` に出す。
- 処理済みとキャンセルが競合した場合は自動更新せず停止する。
- EMS番号、商品コード、数量が欠損した供給は引当へ入れない。
- 商品コードが連結または解析不能の場合は推測分割せず個別診断へ回す。
- 書込前検証に失敗した場合は部分書込をしない。
- 途中エラー時に同じ操作を再実行しても、取置IDと目標数量の洗い替えで二重計上しない。

## テスト方針

純粋計算関数を引当本体から分離し、Node.jsテストで次を固定する。

1. 注文数量から取り置き中数量だけを差し引く。
2. 発送済み、キャンセル戻し、手動解除は注文の確保数量に含めない。
3. 初回登録を二回実行しても取り置き数量が増えない。
4. ④を二回実行しても同じEMS割当が増えない。
5. 処理済み注文だけが発送済みに遷移する。
6. キャンセル注文がキャンセル戻しになり、現物確認前は再引当されない。
7. 現物確認済みキャンセル品が最古の待ち注文へ一度だけ再引当される。
8. 待ち注文がないキャンセル品だけがYahoo戻し候補になる。
9. ⑤を同じ処理IDで再実行してもYahoo移動数量が増えない。
10. `KRBLCM16-02` の過去出荷が今回EMSを消費しない。
11. `OFW300`、`OFW301`、`MZBGD03`、`JNXGD01` の開始前取り置きを今回EMSへ再引当しない。
12. `YMNGD09` が3個到着し1個使用なら余り2個になる。
13. `JMEE167` の10個を7個と3個へ分けて確保できる。
14. `OFW304-1`、`OFW304-2`、`OFW305-1`、`OFW305-2` を別商品として扱う。
15. `POEM65`、`RECIPE42`、注文番号指定在庫を通常FIFOへ混ぜない。
16. `10117375` の注文番号指定数量2を同一注文へ割り当てる。
17. ①のCSV更新で台帳が消えない。
18. 取り置き超過、EMS超過、重複IDで書込前検証が失敗する。
19. 既存のProject_24回帰テストがすべて成功する。

## 移行と後方互換

1. 現行シートとスクリプトをバックアップする。
2. 初回登録が完了するまでは現行④を実行しない。
3. 初期台帳の不変条件を検算する。
4. 新④をプレビュー実行し、現行結果との差分を商品コード・注文番号単位で確認する。
5. 問題がなければ新台帳を正として切り替える。
6. 既存の `取り置き数` 列と消込台帳の `取り置き` メモは読み取り専用の移行参考にし、切替後の計算には使わない。
7. 既存の引当履歴は監査用に保持するが、切替後の注文別確保数量には重ねて加算しない。

## 完了条件

- 初回入力後、通常の①と④だけで取り置き台帳が維持される。
- Yahoo即納在庫を初回棚卸ししなくても注文別取り置きが正しく計算される。
- 過去の注文・出荷が今回EMSを推測消費しない。
- 取り置き中数量と注文数量が商品行単位で一致する。
- キャンセル戻しが現物確認後に最古の待ち注文へ優先再引当される。
- 待ち注文がない現物だけがYahoo戻し候補になる。
- 箱別に `供給 = 取り置き中 + 発送済み + 戻し未処理 + 在庫なし確定 + Yahoo移動済み + 余り` が成立する。
- 実例回帰テスト、既存Project_24テスト、全Project_24 JavaScript構文検査が成功する。
- 不一致がある場合は自動補正や部分書込をせず、対象行を具体的に表示する。

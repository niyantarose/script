# Project_24 取り置き登録高速化・分割行表示整合 設計仕様

作成日: 2026-07-24
対象: Project_24（引当ファイル）

## 1. 背景

受注番号 `10117477`・SKU `MRBLUE41` は、現在の有効な引当が1個であるにもかかわらず、受注明細上では発送済みの2個行と未発送の1個行の両方に「確保済み1個」と同じ現行EMS番号が表示されている。

原因は、引当計算が受注番号・SKU単位で求めた現在確保数を、同じキーを持つ分割行すべてへ繰り返し表示しているためである。台帳上の有効引当数は1個で正しくても、画面上は複数行がそれぞれ確保されたように見え、確認作業で誤解が生じる。

また、`📋 取り置き登録を更新` は候補行ごとに入力規則を設定・削除している。候補行数に比例してSpreadsheetサービス呼び出しが増え、現在のデータ量ではApps Scriptの最大実行時間を超える場合がある。

なお、発送済みの分割行をP列需要へ含めていた問題は別修正 `ef5be8d` で解消済みである。本設計ではP列・引当台帳・供給数量の計算方法を変更しない。

## 2. 目的

1. 受注明細の現在引当表示を、分割行の実際の未発送数量と一致させる。
2. 発送済み行を現在の引当済み行に見せない。
3. 受注番号・SKU単位の有効確保数と、未発送行に表示する確保数の合計を一致させる。
4. 取り置き登録の入力規則設定を一括処理し、通常のデータ量で6分以内に更新を完了させる。
5. 数量の正本である引当台帳、P列、EMS数量を変更せず、表示と更新性能のみを修正する。

## 4. 数量の正本

数量の正本と優先順位は次のままとする。

1. 引当台帳の有効行
2. P列の現在引当先
3. EMSリストの供給数量
4. 受注明細・取り置き登録は上記を表示する画面

受注明細の行別表示は新しい数量を作らない。受注番号・SKU単位で計算済みの現在確保数を、同じキーの未発送行へ配分して表示するだけとする。

## 5. 分割行の表示ルール

### 5.1 対象キー

同一注文内の `受注番号 | SKU` を1グループとする。現在確保数、確保内訳、現在EMS番号はこのキー単位で取得する。

### 5.2 発送済み判定

既存共通関数 `引当_行出荷済み_(row, map)` を使用する。行単位の発送日または既存の発送済み判定に該当する行は、現在引当の表示対象から除外する。

発送済み行では、現在引当に関する次の列を空欄にする。

- 確保済み数
- 不足数
- 確保内訳
- 現在EMS番号

発送履歴そのものは引当台帳に残す。過去にどの箱から発送したかを消す処理は行わない。

### 5.3 未発送行への配分

同じキーの未発送行を受注明細の上から順に処理する。計算済みの現在確保数を、各行の注文数量を上限として古い行から配分する。

各行の計算式は次のとおり。

```text
行確保数 = min(その時点の未配分確保数, 行注文数量)
行不足数 = max(行注文数量 - 行確保数, 0)
次の未配分確保数 = max(未配分確保数 - 行確保数, 0)
```

現在確保数が未発送注文数量を超える異常値の場合でも、表示上は未発送注文数量までに制限する。超過分を別行へ表示したり、台帳を自動修正したりしない。既存の整合性チェックで検出対象とする。

### 5.4 確保内訳とEMS番号

- 行確保数が0の未発送行は、確保内訳と現在EMS番号を空欄にする。
- 未発送行が1行だけの場合は、現在の受注番号・SKU単位の確保内訳とEMS番号をそのまま表示する。
- 未発送行が複数あり、行確保数が1以上の場合は、現在の確保内訳を表示し、末尾に `［この行N／SKU合計M］` を付ける。
- EMS番号が複数ある場合は既存の表記を保持し、行ごとに架空の箱別数量へ分解しない。

確保内訳の表示は監査用の補助表示であり、行別の新しい台帳レコードを作らない。

### 5.5 不変条件

キーごとに次を満たすこと。

```text
発送済み行の行確保数合計 = 0
未発送行の行確保数合計 = min(現在有効確保数, 未発送注文数量合計)
各行の 0 <= 行確保数 <= 行注文数量
各未発送行の 行確保数 + 行不足数 = 行注文数量
```

## 7. 取り置き登録の入力規則高速化

### 7.1 現在の問題

候補行を1行ずつ走査し、各行に対して `getRange().clearDataValidations()` または `setDataValidation()` を呼んでいる。このため数百行で数百回以上のSpreadsheetサービス呼び出しが発生する。

### 7.2 一括設定方式

候補データをメモリ上で作成した後、対象行数と同じ長さの入力規則二次元配列を作る。

- 棚確認列用配列
  - 通常の候補行: 既存の棚確認ルール
  - 返品・即納など棚確認不要行: `null`
- 処理列用配列
  - 返品処理対象行: 既存の返品処理ルール
  - その他: `null`

対象列ごとに `setDataValidations()` を1回だけ実行する。行単位のRange取得と入力規則設定は禁止する。

### 7.3 古い入力規則の削除

- 現在の候補行範囲は、二次元配列内の `null` で不要な規則を同時に削除する。
- 前回より候補行が減った場合は、前回末尾から今回末尾までの管理対象列だけを一括で `clearDataValidations()` する。
- シート全体の入力規則は削除しない。
- 見出し、操作ボタン、管理対象外の列には触れない。

### 7.4 保持する内容

高速化前後で次を保持する。

- お取り置きメモなど既存の手入力値
- 自動色分けと条件付き書式
- 列幅、固定行、フィルタなど既存の画面設定
- 棚確認と返品処理の選択肢・入力拒否設定

入力規則の生成方法だけを変え、値の保存キーや復元方法は変更しない。

### 7.5 計測ログ

更新処理の主要区間を実行ログへ記録する。

- データ収集
- 値の一括書き込み
- 入力規則の一括設定
- 書式反映
- 合計時間

ログは性能確認用であり、シートへ新しい列は追加しない。

## 8. エラー時の扱い

- 行別表示の配分に必要な注文数量が数値化できない場合、そのキーの書き戻しを中止し、受注番号・SKU・行番号をエラーへ含める。
- 入力規則配列の行数が書き込みRangeの行数と一致しない場合は、入力規則を設定せず例外にする。
- 表示更新の失敗を理由に引当台帳、P列、EMS供給数量を変更しない。
- 既存の手入力値は、入力規則設定の失敗時も消去しない。

## 9. テスト仕様

### 9.1 分割行配分の単体テスト

純粋関数として行別配分を検証する。

1. 発送済み2個行と未発送1個行、現在確保1個
   - 結果: 発送済みは空欄、未発送は確保1・不足0
2. 未発送2個行と未発送1個行、現在確保1個
   - 結果: 上の行が確保1・不足1、下の行が確保0・不足1
3. 未発送2個行と未発送1個行、現在確保3個
   - 結果: 上の行が確保2、下の行が確保1
4. 未発送合計3個、現在確保4個
   - 結果: 表示確保合計は3個で打ち止め
5. 現在確保0個
   - 結果: 全未発送行の確保内訳・EMS番号は空欄

### 9.2 入力規則計画の単体テスト

1. 通常候補行は棚確認ルールを持つ。
2. 返品行は処理列ルールだけを持つ。
3. 即納など棚確認不要行は両方 `null` になる。
4. 候補0行でも古い管理対象範囲だけを削除できる。
5. 規則配列の行数が候補行数と一致する。

### 9.3 回帰テスト

- `tests/project24_torioki.test.js`
- `tests/project24_zenken.test.js`
- `tests/project24_daniel.test.js`

既存のP列テスト、全件再計算テスト、ダニエル関連テストを通す。

### 9.4 実シート確認

1. `📋 取り置き登録を更新` が6分以内に完了する。
2. 受注10117477のMRBLUE41が本仕様6章の表示になる。
3. P列が `10117477:1` のままである。
4. 引当台帳の有効数が1個のままである。
5. お取り置きメモと自動色分けが更新前後で保持される。
6. 棚確認・返品処理のプルダウンが従来どおり動作する。

性能目標は、現在の実データで取り置き登録単独更新を120秒以内とする。120秒を超えても6分以内なら機能上は合格とし、ログを基に追加高速化を別課題で判断する。

## 10. 実装順序

1. 行別配分の純粋関数と失敗するテストを追加する。

3. 入力規則計画の純粋関数と失敗するテストを追加する。

5. Nodeテストと構文検査を実行する。
6. Project_24を安全pushする。
7. 実シートで10117477、P列、台帳、メモ、色分け、実行時間を確認する。

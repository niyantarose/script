# 取り置き登録「出てくる注文は全行表示」設計 (2026-07-20)

## 背景

2026-07-20 に取り置き登録シートを「全行表示化」した（即納行を水色・表示専用で追加、
判断済み〈出荷済み/未着/予約〉も隠さない、記憶は棚確認セルへ一本化）。本番反映済み
（v3ブランチ 1bf0162）。

運用中に**残る穴**が判明した（実例: 受注 **10116625** 八谷 結）：

1. 棚に来ている（部分在庫）のに**取り置き登録シートに行が出てこない商品**があり、数量を
   入れられなかった。原因はその行が **受注ステータス「予約中」で入荷スタンプが無い**ため、
   証拠ベースの絞り込み（`取り置き_登録絞り込み_`）で「まだ届いていない予約」と判定されて
   非表示になっていた。実際は早着して棚にあるのにスタンプが無いだけ。
2. `区分_` は `即納 / 取り寄せ / 指定なし` を返し、**指定なし**行は収集ループで捨てられていた。
   ユーザー確認では「指定なしはほとんど即納」。
3. **台湾・中国（別ルート）** は `引当_別ルート判定_` で候補収集から丸ごと除外されていた。
   だが GoQ で「部分在庫」になった台湾・中国は**別ルートで現物が到着した合図**であり、
   見えないと「届いているのに気づけない」。混在注文（韓国＋台湾中国）では、韓国側が
   引き当たり台湾中国側が入荷日未入力で不足 → 注文が「部分在庫」に入るので、注文自体は
   アクティブとして拾える（[引当.js:1644](../../../Project_24/引当.js) の分類ルール）。

ユーザー要望（原文趣旨）:
> 出てくる注文は、即納でも取り寄せでも注文内容を全部出したい。部分在庫は表示したい。
> 出荷GOはもう確定で荷物が届いているから表示不要。指定なしはほとんど即納。
> 台湾中国も GoQ で部分在庫になっていたら別ルートで到着した合図だから出してほしい。

## 目的

**このシートに1行でも出てくる注文（＝棚に何か来ている「アクティブ注文」）は、その注文の
受注明細を全行表示し、注文と箱（現物）を 1:1 で突き合わせられるようにする。** 取りこぼしゼロ。

## 非目的（変更しないこと）

- **純粋な予約だけの注文（棚に何も来ていない）は従来どおり非表示。** 7/18 の「証拠ベース表示」の
  巻き戻しはしない。証拠ゲートは**注文単位**で維持する（注文が出るか出ないかの判定は今までどおり）。
  本設計が変えるのは「出てきた注文の“中”で、これまで隠していた行も出す」＝**注文内の網羅性**だけ。
- 台帳の構造・④の引当ロジック・全件再計算・別ルート済数量の仕組みは変えない。

## 仕様

### アクティブ注文の定義（変更なし）

受注番号（ban）が次のいずれかのソースに載る注文＝アクティブ:
入荷スタンプあり（着済スタンプ）／部分包装／④の各一覧（出荷可能・出荷GO未入金・部分在庫・
希望日待ち）。混在注文の台湾中国不足は「部分在庫」に入るので、この定義で拾える。

### 行の種類ごとの表示ルール

アクティブ注文の受注明細を全行走査し、種類ごとに次の扱いにする。分類は純粋関数
`取り置き_明細区分_(kbn, 別ルート)` に集約する（`kbn`=`区分_` の結果、`別ルート`=`引当_別ルート判定_`）。

| 種類（分類結果） | 元の判定 | 扱い | 取置ID接頭辞 |
|---|---|---|---|
| **即納** | `区分_`＝即納、または**指定なし**（ほぼ即納のため即納扱い） | 水色 `#cfe2f3`・**表示専用**（数量入力不可＝反映時エラー） | `即納\|` |
| **取り寄せ** | `区分_`＝取り寄せ かつ 非別ルート | **チェック可能**（数量入力可）。**予約中で証拠なしも隠さない** | `INIT\|` |
| **別ルート** | `引当_別ルート判定_`＝true（台湾/中国。区分に関わらず優先） | 橙系 `#fce5cd`・**表示専用**。旧入荷日欄に手入力入荷日を出し、空なら要対応欄に「入荷日を受注明細に入れて確保」 | `別ルート\|` |
| （出荷GO） | 受注ステータスに「出荷GO」を含む | **非表示**（従来どおり。確定・到着済み・発送作業中） | — |

`区分_`＝指定なし かつ 別ルート の行（台湾中国で選択肢が取り寄せ/即納でない）は、**別ルートが優先**
（別ルートは区分より先に判定する）。

### 変更点の要約

1. **`取り置き_登録絞り込み_`**: 除外条件を「出荷GO のみ」に減らす（「予約中で入荷証拠なし → 非表示」の
   除外を**やめる**）。候補は既にアクティブ注文の行だけなので、この緩和で純粋予約注文のノイズは増えない
   （影響は「アクティブ注文の中の予約中行」だけ）。
2. **`取り置き_明細区分_`（新・純粋関数）**: 受注明細1行を `即納 / 取り寄せ / 別ルート / 対象外` に分類。
   別ルートを最優先、次に即納（指定なし含む）、取り寄せ。`取り置き初期登録を作成本体_` の収集ループは
   これを使い、即納・別ルートを各枠へ、取り寄せを従来どおり `orders` へ振り分ける。
3. **`取り置き_別ルート行を付与_`（新・純粋関数）**: `取り置き_即納行を付与_` と同型。アクティブ注文の
   台湾中国行を橙系・表示専用（判定=別ルート）で差し込む。旧入荷日は確保状態の目印として持つ。
4. **反映ガード拡張**: `取り置き_初期確定計画_` の表示専用ガードを `即納|` に加えて `別ルート|` にも適用
   （数量が入っていたらエラー。別ルートは「入荷日を受注明細に入れて確保」と案内）。
5. **書式・入力規則**: `取り置き_登録行書式を更新_` に判定=別ルートの色を追加。棚確認プルダウンは
   即納と同様に別ルート行では付けない。

### 表示・入力の細部

- 予約中で証拠なしの取り寄せ行が出ても、`取り置き_棚確認判定_` は証拠が無いので**要棚確認（黄色）に
  しない**＝普通の白い行。受注ステータス列に「予約中」が見え、棚にあれば数量を入れる／無ければ空。
- 台湾中国（別ルート）行の確保は**従来どおり受注明細に入荷日を手入力**する（別ルート済数量の仕組み）。
  この画面で数量を入れると台帳の開始前在庫と二重引当になるため入力不可（表示専用）。旧入荷日欄で
  「確保済みか未対応か」を一目で判断できる。
- 反映ガード（即納行・棚確認と数量の矛盾）は現状のまま。

## テスト（TDD, `tests/project24_reservation_workflow.test.js`）

1. `取り置き_明細区分_`: (取り寄せ,false)→取り寄せ / (即納,false)→即納 / (指定なし,false)→即納 /
   (取り寄せ,true)→別ルート / (指定なし,true)→別ルート。
2. `取り置き_登録絞り込み_`: 予約中で証拠なしの行が**残る**、出荷GO行は**除外**（既存2テストを新仕様へ更新）。
3. `取り置き_別ルート行を付与_`: アクティブ注文の台湾中国行だけ判定=別ルート・数量入力欄空・
   旧入荷日を反映・要対応に確保案内、非アクティブ注文には付けない、メモは洗い替えで引き継ぐ。
4. 反映ガード: 別ルート行に数量→エラー、空なら台帳へ登録しない。
5. 既存テスト（即納表示専用・判断済み全行表示・棚戻し・台帳確保重ね）は全てGREENのまま。

## 1. 背景・目的

EMS同期データを廃止し、韓国側で手動入力する2シート
**「発注リスト大邱データ」「EMS大邱作業データ」** を新設した。これらの新規データを、本体の
**「発注」「EMSリスト」** へ **ボタンで追記** できるようにする。

- 既存の「発注」「EMSリスト」のデータと数式は**一切変更しない**。最終データ行の下に追記する。
- 重複（すでにある行）は送らない。
- EMSリストへ追記したあとは購入No(F列)が空なので、続けて購入No補完で埋める。

## 3. 機能1: 発注リスト大邱データ → 発注（ボタン）

関数（仮）: `大邱_発注へ転送()`

1. 「発注」既存行から重複判定セットを作る:
   `key = 購入No(G).trim() + '|' + normCode_(商品コード(L))`
2. 「発注リスト大邱データ」を上から走査。発注NO(F)と商品コード(K)が両方ある行のみ対象。
   `key = 発注NO.trim() + '|' + normCode_(商品コード)` がセットに無ければ「新規」。
3. 新規行を、発注の**最終データ行の下**へまとめて追記（既存の `H2E_findNextAppendRow_` 同等のロジックを流用）。
4. 列対応（大邱 → 発注）:
   - F→G（購入No。**値そのままコピー**。`Wata…` が来たらそのまま）
   - H→I（業者）/ I→J（商品名）/ J→K（オプション）/ K→L（商品コード）/ L→M（発注数量）
   - N→O（品目）/ O→P（重さ）/ P→Q（価格）/ T→U（決済方法）/ U→V（決済日）
   - C→D（入荷日→入荷日。`購入No補完`の照合に効くので必ず入れる）
   - 予約(H)は対応元が不明確なので**空のまま**（必要なら後で追加）。
   - 小計R・支払金額T・照合キーZ・消込判定AA・韓国系W/X は**書かない**（数式が自動で伸びる）。
5. 追記行の **Y列(EMS発送数)** に SUMIFS を入れる:
   `=SUMIFS('EMSリスト'!$J$7:$J,'EMSリスト'!$F$7:$F,$G{row},'EMSリスト'!$I$7:$I,$L{row})`
   （既存 `発注_EMS発送数数式を一括修正()` と同型。追記後に新規行ぶんだけ書き込み）
6. 実行前に「○件を追記します（プレビュー）」を `ui.alert(YES/NO)` で確認。
7. 完了トースト: `発注へ追記: N件 / 重複スキップ: M件`。

### エッジケース
- **Wata表記**: 旧 `Wata260402_01`(yymmdd) と 大邱 `20260402_01`(yyyymmdd) は文字列が違うため、初回は既存ぶんが「新規」に見えうる。→ プレビューで件数と先頭サンプルを見せ、ユーザーが中止できるようにする。

## 4. 機能2: EMS大邱作業データ → EMSリスト（ボタン）

関数（仮）: `大邱_EMSリストへ転送()`

1. 「EMSリスト」既存行から重複判定セットを作る:
   `key = normTrack_(EMS番号 M) + '|' + normCode_(商品コード I) + '|' + 数量(J).trim()`
2. 「EMS大邱作業データ」を走査。**D列(EMS番号)が空でない行のみ**対象。
   `key = normTrack_(EMS番号 D) + '|' + normCode_(短縮後コード) + '|' + 数量(I).trim()` がセットに無ければ新規。
3. 新規行を EMSリストの最終データ行の下へ追記。
4. 列対応（大邱 → EMSリスト）:
   - A→B（入荷日）/ B→C（EMS発送日）/ D→M（EMS番号）/ H→I（商品コード※下記正規化）/ I→J（数量）/ K→K（品目）
   - A列(No.)=連番（既存 `H2E_getNextEmsNo_` 同等）/ G列(ステータス)=`未着` / F列(購入No)=空
5. **商品コード正規化**: `codeKeys_(rawCode)` の最後の要素（短縮形があれば短縮形）を採用。
   - `KRSJCM03-0506_06` → `KRSJCM03-06`、`KRSJCM03-0506S` → そのまま。
6. 追記後、**自動で機能3（購入No補完）を実行**して新規行のF列を埋める。
7. 完了トースト: `EMSリストへ追記: N件 / 重複スキップ: M件 / 補完: K件`。

## 5. 機能3: 購入No補完

- 既存 `EMSリスト_購入No自動補完()` を**そのまま流用**（照合先は「発注」シートのまま）。
- 機能2の最後に `EMSリスト_購入No自動補完(true)`（silent）を呼ぶ。
- 手動ボタンも残す。

## 6. メニュー（手動運用）に追加

`onOpen` 系（`Excel同期メニューを追加_` または該当メニュー）へ:
1. 発注リスト大邱データ → 発注（`大邱_発注へ転送`）
2. EMS大邱作業データ → EMSリスト（`大邱_EMSリストへ転送`）
3. 購入No補完（`EMSリスト_購入No自動補完`）
4. EMS発送数の数式を一括修正（`発注_EMS発送数数式を一括修正`）

## 7. 流用する既存ヘルパー

- `normCode_(v)`: trim/大文字化/`_`→`-`
- `codeKeys_(code)`: セット巻数の短縮形を返す
- `normTrack_(v)`: EMS番号の空白除去・大文字化
- 追記行さがし / 連番さがし: `H2E_findNextAppendRow_` / `H2E_getNextEmsNo_`（必要なら汎用化して両方で使う）

## 8. 安全策・非機能要件

- 既存データ・数式は read のみ。書き込みは**最終行の下**への追記とY列の新規行のみ。
- `LockService.getDocumentLock()` で多重実行を防ぐ。
- ヘッダー行・空行はスキップ。発注NO/EMS番号/商品コード等の必須が欠けた行はスキップ（件数を報告）。
- 各転送は実行前にプレビュー確認（最低でも機能1は必須、機能2も件数確認）。

## 9. 検証方法（GASは自動テスト不可のため手動）

- 機能1: 大邱に1件だけ新規行を足して実行 → 発注の最終行の下に1件追記され、Y列にSUMIFSが入り、再実行で「重複スキップ」になることを確認。
- 機能2: 大邱EMSにD列EMS番号入りの新規1件 → EMSリストに追記され、商品コードが短縮形になり、補完でF列(購入No)が埋まることを確認。分割発送（同コード別EMS番号）が別行として両方残ることを確認。
- 既存行・数式が変化していないことを確認（小計/支払金額/消込判定）。

## 10. 実行順の推奨
購入No補完は「発注」を照合先にするため、**先に機能1（発注へ追記）→ 後に機能2（EMSへ追記＋補完）** の順で運用すると、新規注文の購入Noも正しく埋まりやすい。

## 11. 確定事項 / 残課題
- 入荷日: 大邱C列 → 発注D列（確定。補完の照合に使用）。
- 予約(発注H): 対応元不明のため空（残課題：必要なら対応元を指定）。
- ステータス既定値「未着」: 確定。
- 商品コード正規化（短縮形採用）: 確定。

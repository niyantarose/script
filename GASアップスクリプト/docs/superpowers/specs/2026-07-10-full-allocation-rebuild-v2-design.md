# 全件検算と安全な再構築(改訂版v2) 設計

日付: 2026-07-10 / 対象: Project_24(引当ファイル) / 旧版: `2026-07-10-full-allocation-rebuild-design.md`(不採用)

## 旧版からの方針転換

旧版は「P列・引当履歴・入荷日を信用せず、新エンジン(全件再計算.js)で全件を再構築して書き戻す」設計だった。これを採用しない理由:

1. **第2エンジン問題**: 旧版の反映処理はP列・履歴・入荷日・行色を直接書き換えるが、これらは②引き当て実行が管理するデータ。新エンジンが②の意味論(色ルール・入荷日ズレ自動修復・部分在庫判定など)を完全に複製しない限り、プレビューで承認した状態は②の初回実行で崩れる。棚卸設計で避けた「正が2つ」問題の引当版になる。
2. **過去の帰属は復元でなく推定**: GoQ発送済みを到着日FIFOで過去の箱へ再配分しても、EMS到着日は追跡同期で後から動く(実例7/9→7/10)し、Yahoo自由在庫から出た出荷はEMS箱を通っていない。推定結果を「記録」として書き込むのは危険。将来の正しさには②+締め直しで足りる。
3. 再構築の骨格は実装済み: 🔄全リセット → 現物確認 → 📦棚卸箱 → ② → 🔎整合チェック。

ただし旧版には採用価値のある部品が3つある。本改訂版はそれだけを既存フローへ移植する。

## 目的(本改訂版で作るもの)

1. **🔄全リセットの事前バックアップ**: リセット前に引当ファイルをDriveで丸ごとコピーする。唯一の非可逆操作(引当履歴の消去)を可逆にする。
2. **🧮全件検算レポート(読み取り専用)**: 商品コードごとに EMSリスト×消込台帳×受注明細×Yahoo自由在庫 を突き合わせ、「発送済み消費・現在受注・過去便の対応が正しいか」を1枚で見えるようにする。**何も書き換えない。**
3. **出荷済みの重複排除の共通関数化**: 同じ発送が表記ゆれ(受注番号タグ・a/b枝番)で台帳に二重にいる場合に1件として数える規則を関数にする。

## 非目標

- P列・引当履歴・受注明細への自動書き込み(レポートは読み取り専用)
- ②引き当て実行・🔎整合チェック・棚卸の挙動変更(重複排除の適用は新レポートのみ)
- Yahoo CSV・Yahoo商品在庫の更新
- 定期トリガー

## 機能1: 全リセットの事前バックアップ

対象: `Project_24/入荷日チェック.js` の `引当データの全リセット本体_`。

- 「リセット」入力確認の後・ロック取得の前に、引当ファイル(アクティブなスプレッドシート)を `引当ファイル_リセット前_yyyyMMdd_HHmmss` の名前でDriveへ丸ごとコピーする。
- 保存先は元ファイルと同じフォルダ(親が取れない場合はマイドライブ直下)。
- コピー作成に失敗したら**何も消さずに中止**し、エラーを表示する。
- 完了ダイアログにバックアップ名を表示する。
- 発注共有ファイルはリセットで変更しないためバックアップ対象外。

## 機能2: 全件検算レポート

メニュー「🏠 メイン引当」に `🧮 全件検算レポート(読み取り専用の突き合わせ)` を追加。実行すると「全件検算」シートを再生成する。受注明細・EMSリスト・台帳・発注共有には一切書き込まない。

### データ源と正規化

| 源 | 使う列 | 正規化 |
|---|---|---|
| 発注共有EMSリスト(6行目見出し) | ステータス列/商品コード/数量/EMS到着日 | `normCode_`(受注番号タグ除去のみ。末尾1字は落とさない) |
| 消込台帳(出荷済み行=`消込台帳_出荷済み行_()`) | 受注番号/商品コード/SKU/個数/入荷日 | `normCode_(SKU優先)`+末尾A/B落とし(棚卸の需要側と同じ) |
| 受注明細(取り寄せ・個数>0・台湾/中国ルート除外) | 商品コード/SKU/個数/項目選択肢/入荷日/商品名 | 同上 |
| Yahoo在庫CSV(最新) | sub-code末尾`a`のquantity | sub-codeから末尾aを外して`normCode_` |

- 即納行は対象外(即納の在庫はYahooが正で、EMS箱と突き合わせない)。
- `b`(お取り寄せ販売枠)は在庫として数えない。
- 日付照合は `ymd_` で正規化した文字列一致(🔎整合チェックと同じ)。
- 既知の限界: 商品コード自体が素でA/Bで終わる商品(例 EBS1504B)は、受注側の末尾落としで別コードに割れて2行に見えることがある。誤警告にはならない(行が分かれるだけ)。

### 商品コードごとの集計列

対象コード = EMSリスト(全ステータス) ∪ 取り寄せ需要 ∪ 台帳出荷済み に現れるコード。Yahooだけにあるコードは行にしない(約2.6万コードのノイズ防止)。

| 列 | 定義 |
|---|---|
| EMS到着済 | ステータス=到着済 の数量計 |
| EMS反映済 | ステータス=在庫反映済み の数量計 |
| EMS未着 | それ以外のステータスの数量計 |
| 出荷済(到着箱) | 台帳出荷済みのうち入荷日がそのコードの**到着済**箱の到着日と一致する数量計 |
| 出荷済(過去便) | 同・**在庫反映済み**箱の到着日と一致する数量計 |
| 出荷済(箱不明) | 入荷日なし、またはどの箱の到着日とも不一致 |
| 確保済(到着箱) | 未出荷取り寄せで入荷日が到着済箱と一致 |
| 確保済(過去便) | 同・在庫反映済み箱と一致 |
| 確保済(ズレ) | 入荷日ありだがどの箱とも不一致(🔎の対象) |
| 待ち | 入荷日なしの未出荷取り寄せ |
| 到着済残 | EMS到着済 − 出荷済(到着箱) − 確保済(到着箱) |
| Yahoo a在庫 | CSVの末尾aのquantity計(参考。CSVが読めない時は「—」) |

台帳出荷済みは集計前に**機能3の重複排除**を通す。

### 判定(上から先勝ち)

1. `到着済残 < 0` → **⚠️超過消費** — 箱の個数以上に消費扱い。幽霊スタンプ/記録漏れの疑い。詳細は🔎整合チェック。
2. `確保済(ズレ) > 0` → **⚠️入荷日ズレ** — 🔎整合チェックで行単位を確認。
3. `待ち > EMS未着 + max(0, 到着済残)` → **📦供給不足** — 見えている供給で待ち注文を賄えない。棚卸箱の候補。
4. `到着済残 > Yahoo a在庫` → **ℹ️箱残>Yahoo** — 締め前の便なら正常(箱の余りは締めでYahooへ反映される)。締めたはずなら未記録出荷の疑い。
5. `到着済残 < Yahoo a在庫` → **ℹ️EMS外在庫** — Yahoo自由在庫がEMS由来より多い。免税店買付等があるなら正常。
6. どれでもない → **OK**

Yahoo CSVが読めない場合は判定4・5を出さず、タイトル行に「Yahoo照合なし」と注記して続行する。

### 出力シート「全件検算」

- 1行目: タイトル(実行日時 / 使用CSVファイル名 / 判定別件数)
- 2行目: 見出し(固定表示)。3行目以降: ⚠️→📦→ℹ️→OK の順、同判定内はコード順。
- 行背景: ⚠️=赤(`色_赤`)、📦=黄(`色_黄`)、ℹ️/OK=なし。
- 一覧の下に凡例(各判定の意味と対処先: 🔎整合チェック / 📦棚卸箱 / 便の締め)。
- 実行は `直列_` でロック(他の書き込み処理と同時に走らせない)。

## 機能3: 出荷済みの重複排除

`Project_24/消込台帳.js` に追加:

```
出荷済み重複排除_(rows) // rows: 消込台帳_出荷済み行_() の戻り値
// キー = 受注番号 + '|' + 基底コード(SKU優先・タグ除去・末尾A/B落とし)
// 同キーが複数 → 数量が最大の1件だけ残す(同じ発送の表記ゆれとみなす)
```

- 適用は**全件検算レポートのみ**。②・🔎の消費計算には適用しない(実測で調整済みの挙動を変えないため)。将来🔎の幽霊判定のノイズが問題になったら適用を検討する。

  - `全件検算レポート()` / `全件検算レポート本体_()`: シート読み込み→集計→描画

## テスト

### 代表ケース

1. YahooCSV集計: 末尾`a`のみ加算・`b`無視・引用符内カンマと`""`エスケープ・壊れた行スキップ・sub-code無し在庫の検出
2. 重複排除: 同一受注番号でタグ付きコード/a枝番SKUの表記ゆれ→1件(数量最大)。受注番号違い・基底コード違いは残す
3. 超過消費: 到着済10・出荷済(到着箱)8・確保済(到着箱)4 → 残-2 → ⚠️超過消費
4. 入荷日ズレ: 入荷日がどの箱とも不一致 → ⚠️入荷日ズレ
5. 供給不足: 待ち5・未着2・到着済残0 → 📦供給不足
6. 箱残>Yahoo: 到着済5・消費0・Yahoo a=2 → ℹ️箱残>Yahoo
7. EMS外在庫: 到着済残0・Yahoo a=3 → ℹ️EMS外在庫
8. 対象外の除外: 台湾/中国ルート・即納・`b`枝番のSKUが基底コードへ丸まる
9. 過去便の分類: 入荷日が在庫反映済み箱と一致 → 確保済(過去便)へ。到着済残に影響しない

### 不変条件

- レポート実行で受注明細・EMSリスト・消込台帳・引当履歴のセルが1つも変わらない(読み取り専用)
- バックアップ失敗時にリセットが何も消さない

## 運用

- 迷ったらまず🧮を実行して現状把握(何も壊さない)。⚠️が出たら🔎整合チェック→対処。供給不足は📦棚卸箱。
- 全部やり直したい時: 🧮で現状確認 → 🔄全リセット(自動バックアップ) → 現物が残る箱を「到着済」へ → 📦棚卸箱 → ② → 🔎 → 🧮で仕上げ確認。

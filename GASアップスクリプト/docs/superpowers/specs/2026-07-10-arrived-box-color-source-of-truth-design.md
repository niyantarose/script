# 到着済み箱の色判定をEMSステータス基準へ統一する設計

## 目的

受注明細の入荷日が予定日のまま残っていても、EMSリストの「到着済」行でP列（注文番号）に確定された注文は黄色にする。便を締めて「在庫反映済み」へ移った割当は、引当履歴を根拠にラベンダーにする。

## 決定事項

- 黄色の根拠は、現在のEMSリストでステータスが「到着済」のP列確定割当とする。
- P列確定割当には商品コード・数量に加え、EMS到着日・EMS番号を保持する。
- P列確定先の受注明細に古い入荷日がある場合、現在のEMS到着日で訂正する。
- P列確定されていない手入力日付は自動上書きしない。
- 「在庫反映済み」は既存の引当履歴を根拠にラベンダー表示を維持する。
- 同一商品コードというだけで全注文を黄色にはしない。P列の注文番号と数量を優先し、過剰引当を防ぐ。

## データフロー

1. `P列確定マップ_` が「到着済」行だけを読み、注文番号ごとに商品コード・数量・到着日・EMS番号を返す。
2. `引当実行_本体_` が、入荷日不一致でもP列で名指しされた数量を現在便から引き当てる。
3. 現在便で全量が成立した行は黄色にし、入荷日を実到着日、EMS番号を該当箱へ更新する。
4. 在庫反映済みの履歴だけで成立する行は従来どおりラベンダーにする。

## 安全条件

- P列未確定の行、台湾・中国ルート、商品コードが一致しない行は訂正対象にしない。
- 数量はEMS在庫残数とP列指定数の小さい方で制限する。
- 既に今回便の到着日と一致する行を二重に消費しない。

## テスト

- 受注明細7/9、P列確定のEMS到着日7/10なら、現在便として黄色になる。
- 上記行の訂正候補日は7/10、EMS番号はP列元の箱番号になる。
- 「在庫反映済み」行はP列確定マップへ入らず、履歴側のラベンダー判定を維持する。

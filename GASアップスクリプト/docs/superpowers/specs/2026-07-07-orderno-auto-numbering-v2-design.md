# 発注NO自動採番v2＋修復・重複チェック 設計

日付: 2026-07-07 / 対象: Project_19（発注EMSリスト）

## 背景・問題

発注NOは「発注日8桁_カート番号2桁_カート内行番号」で全行ユニークが前提。
EMSリストの照合キー＝`購入No-商品コード`はこの3部構成に依存する。

現状の問題（実行ログ・実データで確認済み）:

1. **onEditタイムアウト**: 採番がonEditチェーン後段にあり、30秒制限で頻繁に不発
   （実行ログ: 過去7日でonEditタイムアウト7回）。行番号なしの素値（例
   `20260704_06`×4行）が残る。
2. **カート番号かぶりを枝番で吸収**: 別業者が同じカート番号を使うと `_06_2` の
   ような枝番を付けて同一カートグループに混ぜてしまう（20260626_06問題）。
3. **日付のみ入力で壊れた番号**: `20260704` と入力すると `20260704_1` が生成され、
   カート1と区別できず、再入力で同じ番号が複数できる。
4. **分割発送**: 同じ注文行を複数回に分けて発送するのは正常運用
   （コード内に明記）。「同じ照合キーが2回出たら重複」では判定できない。

## 仕様

### 1. onEdit採番v2（発注リスト大邱データ F列）

- **onEditチェーン先頭で実行**（番号確定を最優先。後段がタイムアウトしても番号は残る）
- 対象は8桁日付型のみ（`Wata…`等は不変更）。データ行は HACHU_SRC_DATA_START(6行目)以降
- 入力パターン:
  - `日付`のみ → 新カート自動採番（日付内max+1、2桁0埋め、欠番不使用）。
    連続する同日付・同業者(H列)の行は同じカート
  - `日付_カート` → 未使用なら`_1`。同業者使用中なら次の行番号。
    **別業者使用中なら次の空きカートへ繰り上げ＋トースト通知**
  - `日付_カート_行` → 空きならそのまま。使用済みなら上記規則
- 一度付いた番号は変更しない（EMS側が参照するため）
- 罫線・チェックボックスは編集近傍のみ再描画（全域drawGroupBordersをonEditから排除）
- ドキュメントロック tryLock(5000)。取れなければ採番スキップ＋トースト（修復で回収）

### 2. 発注NO修復（メニュー）

全行スキャン→プレビュー→実行:

- 行番号なしの行: 所属カートの空き行番号を付与（同一業者カートの採番済み行は不変更）
- 同一カートに複数業者: 最初の業者ブロックがカート番号を保持。以降のブロックは
  **採番済みでも新カートへ分離**し、旧番号→新番号の対応でEMS大邱/EMSリストの
  参照を追随更新（2026-07-07拡張。旧仕様は報告のみだった）
- 業者名の表記ゆれ（大文字小文字・全半角・空白）は同一業者として扱う
- 完全に同じ番号の重複: 報告のみ（自動では触らない）
- 修復完了時、解消済みの赤背景を自動リセット（チェックが塗った色のみ対象）
- 番号を変えた行は EMS大邱(T列)・EMSリスト(F列) の古い参照を
  商品コード＋数量消し込しで新番号へ再割当。コード不一致（★コピペ等）は
  「要手動」として一覧報告
- 仕上げに既存の `大邱_発注EMS枝番を一括修正(true)` と
  `EMSリスト_購入No自動補完(true)` を実行

### 3. 発注NO重複チェック（メニュー）

書き換えなし・検出のみ:

- 大邱側: 行番号なし / 完全重複 / 複数業者かぶり → F列を赤背景＋一覧報告
- EMSリスト側:
  - 照合キーごとの発送数合計 > 発注数 → 送りすぎ（二重発送疑い）
  - 発注側に存在しない購入Noを参照 → 迷子キー（赤背景）
  - 商品コード未入力（空欄・`★`始まり）→ 一覧報告
  - 同キー複数回でも合計≦発注数なら正常（分割発送）

### 4. 転送前の購入No検証ゲート（追加要件 2026-07-07）

- **大邱→EMS大邱（チェック行を送る）**: 送る行の発注NOが「行番号まで確定済み」かつ
  「シート内で一意」でなければ一覧表示して送信を中止（ロック前とロック後の両方で検証）
- **EMS大邱→EMSリスト（転送）**: 追記する行の購入Noが「行番号まで確定済み」かつ
  「発注リスト大邱データに実在する」ことを確認してから転送。NGなら一覧表示して中止
- どちらも中止時に「発注NO：修復」等の対処法を案内する

## 実装

- 新ファイル `【大邱】発注NO採番v2.js`: 純粋ロジック（Node vmでテスト）＋GAS結線
- `【統合】onEdit(e)管理スクリプト.js`: HACHU_SRC分岐の先頭で採番v2、旧呼び出し撤去
- `【大邱】データ転送.js`: 旧 `大邱発注_onEdit採番_`・`大邱発注_一意Noを採番` を削除
  （`大邱発注_連番管理_`等は枝番・発注シート側が使うため残す）
- `エクセルからデータ取得.js`: メニュー「一意No採番」を「重複チェック」「修復」に差し替え
- 検証: 純粋ロジックをNodeでシナリオテスト（正常系・_06衝突・日付のみ・
  複数行貼り付け・修復・EMS再割当・数量チェック）

# 未着便のP列先行記入＋受注明細「入荷予定」列 設計書

作成: 2026-07-21。Project_24（引当ファイル）＋発注共有EMSリスト。
ユーザー要望「先の便に引き当てをしたい。箱が来る前に把握したい」への回答設計。

## 背景と動機

- EMSリストのP列（注文番号列）＝「この箱行は誰の注文分か」。しかし自動記入は
  **「到着済」行にしか書かない**（P列自動記入.js `P列処理対象EMS_` = status==='到着済'）。
  ②P列確定も到着済のみ（P列確定.js:38）。
- そのため輸送中・発送前の箱は「SKU単位の未着数」（全件検算の未着列）までしか見えず、
  **どの注文の分かは箱が着くまで分からない**。
- 例: JMEE167 未着20個 → 「16個は10117602ほか、4個は在庫行き」を発送された瞬間から見たい。

## 決定事項（2026-07-21 ユーザー確認済み）

1. **対象範囲**: 実EMS番号が付いた未着行**すべて**（発送前・輸送中・ステータス空欄含む）。
   韓国側が中身を後編集しても次の②で自動で引き直されるため許容。
2. **視点**: EMSリスト側（P列先行記入）と**受注明細側（「入荷予定」列新設）の両方**。
3. **方式**: 案A =「素の注文番号＋薄青背景」。接頭辞方式・別列方式は不採用
   （到着時の昇格を無操作にする・既存P列パーサを無傷に保つため）。

## 設計

### 1. 対象行の定義

- **確定対象**（従来どおり・不変）: ステータス「到着済」。②確定・④出荷可能の根拠。
- **計画対象**（新設）: `実EMS番号_()` が真 かつ ステータスが「到着済」「在庫反映済み」以外
  （空欄含む）。
- 「在庫反映済み」行はどのパスでも触らない（履歴）。棚卸合成行・EMS番号なし行は従来どおり対象外。

実装: `P列処理対象EMS_(status)`（確定対象判定）はそのまま残し、
新関数 `P列計画対象EMS_(status)` を追加。行オブジェクトに `対象`（確定）と `計画`（未着）の
2フラグを持たせる。

### 2. 割り当てロジック

- 既存の純粋計算エンジン `P列計画_純計算_`（取り置き計算.js:124）は**無変更**。
- 呼び出し側（発注共有P列計画_）で供給行を**2パス順**に並べて渡す:
  **(1) 確定対象ブロック（シート順）→ (2) 計画対象ブロック（シート順）**。
  到着済供給が先に消費されるため、**到着済行の割当結果は現行と完全一致**（テストで保証）。
  古い注文から順に、到着済で足りない分が未着便へ流れる。
- 全件再計算ブロックSKUの行は未着でも nextP=''（到着済と同じ扱い、P列自動記入.js:383相当）。
- **手書き保護**: 現状、未着行のP列は自動が触らないためユーザーの手書きが残っている。
  導入後は自動が書き替えるため、到着済行と同じ「コード不一致の名指し救済」
  （P列自動記入.js:360-371）を計画対象行にも適用する。コードが注文と一致する行は
  計画が正＝上書き（現行の到着済と同じ思想）。

### 3. 見た目（EMSリスト）

- 計画対象行の記入セル: 背景 **#c9daf8（薄青）**＝予定の印。書式は到着済と同一
  （`10117602` / `10117602:16, 10117731:4`。どの注文にも割り当たらない余り数量は
  従来の到着済と同じく何も書かない＝空欄のまま）。
- 到着済へステータスが変われば、次の②で背景が通常ルール（白／分割・残ありは薄黄#fff2cc）に
  変わり、テキスト無変更のまま確定へ昇格。
- 予定が消えた計画対象行（キャンセル等）は空欄＋背景クリア。
- ♻️「P列を書き直す」のクリア範囲に計画対象行を追加（在庫反映済みは不変のまま）。
  ダイアログ説明文も更新。

### 4. 受注明細「入荷予定」列（新設）

- ヘッダー名「入荷予定」で列マップ（列マップ_）解決。**列が無ければ右端に自動作成**。
  （調査済み: 取込_実行_はヘッダー行を保持しデータ域だけ入れ替えるため、右端の
  カスタム列ヘッダーは取込後も生き残る。値は取込で消えるが次の②で全再生成される。
  入荷日のような退避/復元は不要。列の手動移動もヘッダー名解決なので自由。）
- 値の書式: `M/d(…9766)` = EMSリストのEMS到着日（予測⇒含む）＋EMS番号の数字部末尾4桁。
  - 複数便分割: `7/23(…9766)+7/26(…9011)`。3便以上は先頭2便＋`ほか`。
  - 到着日欠落: `(…9766)` のみ。
- 対象: 韓国取り寄せの未着行（入荷日なし）のみ。着済（入荷日あり）・台湾中国別ルート・
  即納は常に空欄。
- 更新タイミング: ②のP列計画反映と同時に**全書き直し**（予定が無くなった行は空欄化）。
  手入力は想定しない（自動管理列）。
- 全件再計算の韓国派生欄クリアへの追加は**不要**（調査結果: 反映フローは最後に
  引当実行_本体_を再実行し、その中のP列反映funnelがこの列を全幅書き直すため）。

### 5. 変更しない安全境界

- ②P列確定: 到着済のみ読む（P列確定.js:38）— 不変
- ④引当実行・入荷日スタンプ・取り置き台帳: 到着後のみ — 不変
- 全件再計算: 供給= 到着済|在庫反映済み（全件再計算.js:453）— 不変
- **「出荷できる」の物理保証にはどのパスも触れない。先行記入は情報表示＋到着時の段取りだけ。**

### 6. テスト

新規 `tests/project24_pre_arrival_plan.test.js`（node・vm方式は既存テストと同じ）:

1. 到着済優先消費: 到着済+未着が混在しても到着済行の割当は未着なし時と完全一致
2. 未着FIFO: 到着済で満たせない古い注文から未着便に流れる（複数便分割含む）
3. ブロックSKU: 未着行に書かない
4. 手書き救済: コード不一致の手書き名指しは未着行でも保持、コード一致は上書き
5. 昇格シナリオ: 未着→到着済にステータスが変わっても nextP が同値（テキスト書き換えなし）
6. 入荷予定文字列の整形（複数便連結・日付欠落・3便以上の「ほか」）

既存8テストファイルの回帰も全PASS必須。

## 運用ノート

- **pushタイミング**: 全件再計算の反映（残作業A）が完了してから本番push。
  反映フローの最中にコードを差し替えない。
- push手順は従来どおり: git pull → gas_pull_sync.ps1 Project_24 → 実装 → テスト →
  gas_safe_push.ps1 Project_24 → scratchへclasp pullで全ファイル一致検証。素のclasp push禁止。
- 初回導入時: 未着行に一斉に薄青の予定が入る。手書き運用していた行は救済ルール
  （コード不一致のみ保持）で整理される。気になる場合は導入直後に♻️P列を書き直すで
  白紙から引き直せる。
- 発注共有は韓国側と共同編集。P列の薄青セルが増えるのは日本側の計画表示であり、
  韓国側の転送・後編集フロー（重複追記抑止・上書きしない）とは干渉しない。

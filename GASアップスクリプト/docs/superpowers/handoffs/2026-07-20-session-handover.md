## 1. なぜ全件再計算(残作業A)をやろうとしているか — JMEE167 過剰消費バグ

②引当が「**P列確定数量を満たせない: 10117602 JMEE167 指定16/割当15**」で中止（他に JPSNCM61-04W-PG も）。
ワークフロー診断(6エージェント・成功)で確定した真因:

- JMEE167は**4便(6+3+10+20=39個)が同じEMS番号を共有**。過去3箱は締め済み(在庫反映済み)、1箱が到着済20。
- **供給キー `取り置き_供給キー_`（[取り置き計算.js:27-30]）が「EMS番号｜商品コード」だけで箱/便を区別しない** +
  **EMS在庫移動台帳の古い行が消えず累積**（締め済み過去箱の消費が残る）→ 過去箱の消費が現在箱20の供給から
  差し引かれ**実効15** → P列16に届かず中止。全件検算の**余りマイナス(超過消費)**も同現象。
- **Yahoo即納8は無関係**（②はYahoo CSVを読まない／全件検算では比較列だけ）＝ユーザーの「即納8を混ぜるな」は
  正しく、コード上も既に混ざっていない。犯人は**古い移動台帳の累積**。
- 取り置き台帳は JMEE167 = 0行（取り置き中なし）。EMS在庫タブは JMEE167 = 20（正しい）。

**直し方 = 全件再計算の反映**。取り置き台帳・EMS在庫移動台帳を事実から丸ごと作り直して置換（各箱の供給上限内でしか
消費しない→余り≥0が構造保証）、受注明細の韓国派生欄クリア→EMS在庫再生成→**④をclearCurrentPでP列全再生成**（[全件再計算.js:701,705,708,711]）→②の中止が解消。まさに全件再計算が設計された目的。

### ⚠️ 反証で判明した重要注意（反映後のJMEE167の帰結）

反映後、**JMEE167が満量引き当たるとは限らない**。2択:
- **未引当(情報)** = 実供給が本当に足りない → 追加発注・元データ修正
- **ブロックSKU(EMS説明不能余り・[全件再計算.js:380])** = 供給39が需要で説明しきれない**過剰供給** →
  過去箱を説明する発送済み/即納/棚確認データを突合修正 → 「🧹 全件再計算のブロックSKUをクリア」で解除して再引当
- **どちらでも「②全体の中止」は解消する**（1商品が情報表示で残るだけ）。JMEE167は過去箱が多くYahoo在庫8なので
  **過剰供給→ブロックの線が濃い**。数値の細部(-31 と 割当15)は directBan/ownerUsage 等別次元も絡む可能性
  （反証指摘）だが、方向と修正(全件再計算反映)は堅い。成功基準は「JMEE167が16引当」ではなく「**②が中止しない**」。

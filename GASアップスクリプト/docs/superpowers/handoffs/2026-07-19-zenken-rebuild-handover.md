## 2. 全件再計算v3（台帳統合版）とは

仕様書: `docs/superpowers/specs/2026-07-19-full-allocation-rebuild-ledger-design.md`
計画書: `docs/superpowers/plans/2026-07-19-full-allocation-rebuild-ledger.md`

事実データ（実EMS到着・GoQ発送実績・現在受注・Yahoo末尾a）から時系列で全件再計算し、
**取り置き台帳＋EMS在庫移動台帳の置換計画**を作り、P列・入荷日・色は**既存④に再生成させる**。

- 実装: `Project_24/全件再計算.js`（純粋エンジン＋GASアダプタ）
- 消費順: GoQ発送済b(発送日cutoff) → 現在即納 → Yahoo a → **開始前在庫充当** → 現在取寄せ
- 説明できないSKUはブロック＝通常④でも供給0扱い（出荷可能にしない）
- フロー: 🧪プレビュー（サマリ/割当明細/要確認/内部シートのみ書く）→ 入力署名の再確認 →
  2ファイルDriveバックアップ → 旧台帳の監査シート保存 → 台帳一括置換 →
  韓国派生欄(入荷日/EMS)クリア → EMS在庫更新 → ④再実行。途中失敗はガードで通常④停止
- 入荷日復元: 台帳行の「終了理由・メモ」に『元EMS到着日=YYYY-MM-DD』を埋め、
  ④が `全件再計算_台帳到着日_` で読んで受注明細へ復元（引当.js 1405付近）
- 復旧メニュー: 「🔓 全件再計算ガードを解除(復旧用)」「🧹 全件再計算のブロックSKUをクリア」

## 4. 残作業B: 取り置き登録の「全行表示化」（提案済み・ユーザー前向き・正式GOはまだ）

### 背景（実事故 7/19）

受注10117188（鶴原 舞華）は即納2＋取り寄せ2の4行構成なのに、取り置き登録には1行しか出なかった。
原因: ①即納行はそもそも候補に入れない設計 ②WTF-SMJ-28-2b は過去の棚確認判断
（出荷済み/未着のどれか）が DocumentProperties(`取り置き登録_棚確認済み`) に残り、
**注文がまだ部分包装（梱包中）なのに行ごと非表示**が続いた。古い入荷日2026-06-29も残存。

7/17時点は「一度きりの棚卸ツール」だったので非表示＝正解だったが、7/18以降
**取り置き登録だけで運用する日常画面**になったため前提が変わった。隠すのではなく
「見えて、状態が色で分かる」に変えるべき、という提案でユーザーは前向き（正式GO待ち）。

### 仕様案

1. **注文に含まれる全行を表示**（即納も含む）。即納行は水色（受注明細と同じ色言語）・表示専用。
   即納行に数量が入っていたら反映時にエラー（開始前在庫として登録させない＝
   全件再計算のinitialHoldsがEMS供給を誤って食うのを防ぐ）
2. **判断済み（出荷済み/未着/予約）行も隠さない**: 棚確認列に判断を入れたまま表示し、
   条件付き書式（出荷済み=赤太字等）で目立たせる。記憶は「非表示スイッチ」をやめて
   「棚確認列の既定値の引き継ぎ」に単純化 → セルを空にすれば次回から消える
3. 未発売の自動予約行も紫で表示（例外を作らない。多すぎたら後でフィルタ検討）
4. 赤い棚戻し待ち→要棚確認の優先並びと「注文グループを分断しない」は現行維持
